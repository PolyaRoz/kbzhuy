## Фаза 1 — MVP

**Цель:** Пользователь проходит полный флоу вручную: регистрация → онбординг → план на неделю → список покупок → план готовки → видит текущий контейнер.

---

### 1.1 Очистка backend-дублей

**Описание:**
В проекте два места с сервисами: `backend/services/` (корень) и `backend/app/services/`. Нужно выбрать одно, остальное удалить. Проверить, что все импорты в роутах ведут в `backend/app/services/`.

**Результат:**
- Единственное место для бизнес-логики: `backend/app/services/`
- `backend/services/` удалена
- Все роуты импортируют сервисы из `app.services.*`

**Зависимости:** нет

---

### 1.2 Импорт рецептов в базу данных

**Описание:**
Написать скрипт `backend/scripts/seed_recipes.py`, который читает `data/recipes/basic_recipes.json` и `data/nutrition/products.json` и загружает данные в таблицы `recipes` и `ingredients`. Запустить его один раз вручную.

**Результат:**
- Таблица `recipes` содержит базовые рецепты
- Таблица `ingredients` содержит продукты с КБЖУ на 100г
- Скрипт: `python backend/scripts/seed_recipes.py`

**Зависимости:** 1.1

### 1.3 Backend: регистрация и логин

**Описание:**
Проверить и довести до рабочего состояния эндпоинты:
- `POST /api/v1/auth/register` — создать пользователя, вернуть `access_token` + `refresh_token`
- `POST /api/v1/auth/login` — вернуть токены
- `POST /api/v1/auth/refresh` — обновить access_token по refresh_token

Протестировать через `curl` или Swagger (`http://localhost:8000/docs`).

**Результат:**
- Регистрация создаёт пользователя в БД
- Логин возвращает токены
- Refresh обновляет access_token
- Все эндпоинты возвращают корректные HTTP-коды (200/201/401/422)

**Зависимости:** 1.1

---

### 1.4 Backend: онбординг и профиль

**Описание:**
Проверить и довести до рабочего состояния:
- `POST /api/v1/profile/onboarding` — сохранить анкету (пол, вес, рост, цель, аллергии, бюджет, расписание приёмов, плановые отклонения)
- `GET /api/v1/profile` — вернуть текущий профиль
- `PATCH /api/v1/profile` — обновить параметры

Все эндпоинты за JWT-авторизацией. Валидация через Pydantic.

**Результат:**
- Профиль сохраняется и читается корректно
- Поля анкеты соответствуют схеме `Profile` в БД
- Тест через Swagger: создать пользователя → сохранить профиль → получить профиль

**Зависимости:** 1.3

---

### 1.5 Backend: генерация плана питания

**Описание:**
Проверить и довести до рабочего состояния:
- `POST /api/v1/plan/generate` — запустить `meal_planner_service`, создать `MealPlan` + `DayPlan[]` + `Meal[]` на 7 дней
- `GET /api/v1/plan/{plan_id}` — вернуть план со всеми днями и блюдами

`meal_planner_service` берёт профиль пользователя → вызывает `nutri_service` для расчёта КБЖУ → подбирает рецепты из БД → распределяет по дням и приёмам.

**Результат:**
- `POST /plan/generate` создаёт план на 7 дней в БД
- Каждый день содержит 3–5 приёмов пищи
- Суммарные КБЖУ каждого дня ≈ целевым (±10%)
- `GET /plan/{id}` возвращает полный план с рецептами

**Зависимости:** 1.2, 1.4

### 1.6 Backend: список покупок

**Описание:**
Проверить и довести до рабочего состояния:
- `GET /api/v1/plan/{plan_id}/shopping-list` — агрегировать ингредиенты из всех блюд плана, сгруппировать по категориям, вернуть `ShoppingList`
- `PATCH /api/v1/shopping/{list_id}/items/{item_id}` — отметить купленным (`checked: true`)

**Результат:**
- Список содержит все ингредиенты плана, сгруппированные: мясо, молочка, овощи, бакалея
- Дубли агрегированы (напр. два рецепта с гречкой → одна строка с суммой)
- Чекбокс работает и сохраняется в БД

**Зависимости:** 1.5

### 1.7 Backend: план готовки и контейнеры

**Описание:**
Проверить и довести до рабочего состояния:
- `GET /api/v1/plan/{plan_id}/cooking-plan` — вернуть `CookingPlan` с шагами и параллельными группами
- `GET /api/v1/plan/{plan_id}/containers` — список контейнеров с метками, содержимым, местом хранения, инструкцией подогрева

Добавить два отсутствующих эндпоинта:
- `GET /api/v1/containers/current` — контейнер, актуальный прямо сейчас (по текущему времени и расписанию приёмов пищи из профиля)
- `GET /api/v1/containers/today` — все контейнеры на сегодня

**Результат:**
- `cooking-plan` возвращает пошаговый план с параллельными группами
- `containers` возвращает список с метками 1А, 2Б… и инструкциями
- `containers/current` определяет текущий приём пищи по времени и возвращает нужный контейнер
- `containers/today` возвращает контейнеры текущего дня в хронологическом порядке

**Зависимости:** 1.5

### 1.8 Mobile: Auth flow

**Описание:**
Реализовать полный auth flow в приложении:
- Экран регистрации (email + пароль) → `POST /auth/register` → сохранить токены в `authStore`
- Экран логина → `POST /auth/login` → сохранить токены
- Автоматический refresh через axios interceptor при 401
- При наличии токена — пропускать auth экраны, показывать главный

Токены хранятся в `SecureStore` (Expo), не в `AsyncStorage`.

**Результат:**
- Пользователь может зарегистрироваться и залогиниться
- Токены сохраняются между перезапусками приложения
- Expired access_token автоматически обновляется
- Роутинг: нет токена → auth экраны; есть токен → табы

**Зависимости:** 1.3

### 1.9 Mobile: Онбординг

**Описание:**
Подключить 4 шага онбординга к реальному API:
- Шаг 1: физические параметры (пол, возраст, рост, вес, активность)
- Шаг 2: цель (похудение / набор / поддержание)
- Шаг 3: предпочтения (аллергии, нелюбимые продукты, бюджет)
- Шаг 4: расписание приёмов пищи + плановые отклонения (пиво по пятницам)

На последнем шаге → `POST /api/v1/profile/onboarding` → redirect на главный экран.
Если профиль уже есть → пропустить онбординг.

**Результат:**
- Все 4 шага собирают данные и отправляют в API
- Профиль сохраняется в БД
- После онбординга пользователь попадает на главный экран

**Зависимости:** 1.4, 1.8

## Фаза 2 — AI-агент и адаптация

**Цель:** ИИ генерирует персональный план и перестраивает его в ответ на отклонения.

---

### 2.1 AI-агент: tools и инфраструктура

**Описание:**
Реализовать базовую инфраструктуру AI-агента в `backend/app/ai/`:
- `agent.py` — цикл ReAct: system prompt → LLM call → parse tool call → execute → loop
- `tools.py` — реализовать инструменты:
  - `get_user_profile` — читать профиль из БД
  - `calculate_kbzhu` — вызывать `nutri_service`
  - `search_recipes` — искать рецепты в БД по фильтрам
  - `get_storage` — читать текущие запасы
- Интеграция с `anthropic` SDK, модель `claude-sonnet-4-6`

**Результат:**
- Агент может принять запрос, вызвать tools и вернуть результат
- Работает `POST /api/v1/agent/chat` с простым диалогом
- Логирование всех LLM-вызовов (tool calls, токены, время)

**Зависимости:** 1.14 (MVP завершён)

---

### 2.2 AI-агент: генерация плана

**Описание:**
Подключить AI-агента к генерации плана питания:
- `POST /api/v1/plan/generate` опционально принимает `use_ai: true`
- При `use_ai: true` агент оркестрирует: `get_user_profile` → `calculate_kbzhu` → `search_recipes` → `build_meal_plan`
- `build_meal_plan` — новый tool, который вызывает `meal_planner_service` с параметрами от агента

**Результат:**
- Агент генерирует план с учётом полного профиля пользователя
- Качество плана выше, чем детерминистического (разнообразие, учёт привычек)
- Время генерации < 30 секунд

**Зависимости:** 2.1

---

### 2.3 Движок отклонений

**Описание:**
Реализовать полноценную обработку отклонений:
- `POST /api/v1/deviations` — зарегистрировать отклонение (тип, описание, КБЖУ)
- `POST /api/v1/deviations/recalc` — пересчитать остаток недели: агент вызывает `handle_deviation` → `calculate_kbzhu(остаток)` → `update_plan`
- `get_storage` tool учитывает запасы в контейнерах при перестройке

**Результат:**
- "Я съел пиццу (800 ккал)" → план на оставшиеся дни адаптируется
- Агент не "ругает", а перераспределяет КБЖУ
- Изменённые дни помечаются как "адаптировано"

**Зависимости:** 2.2

---

### 2.4 Mobile: экран агента

**Описание:**
Подключить экран "Агент / Чат" к реальному API:
- Чат с историей сообщений → `POST /api/v1/agent/chat`
- Быстрые действия: "Я съел не по плану", "Упрости меню на этой неделе", "Что приготовить сегодня вечером?"
- При отклонении — кнопка "Пересчитать план"

**Результат:**
- Пользователь может общаться с агентом
- Быстрые действия работают в один тап
- Отклонение через чат перестраивает план

**Зависимости:** 2.3

---

### 2.5 Автогенерация задач подготовки

**Описание:**
При создании или обновлении плана автоматически генерировать `PrepTask`:
- За 12–24 часа до приёма: "Переложи контейнер 2Б из морозилки в холодильник"
- За день: "Достань бананы из холодильника для дозревания"

Новый tool агента: `generate_prep_tasks` — анализирует план, места хранения, расписание, создаёт задачи.

**Результат:**
- После генерации плана в БД создаются `PrepTask` с расписанием
- `GET /api/v1/prep-tasks/today` возвращает актуальные задачи
- Главный экран показывает задачи подготовки на сегодня

**Зависимости:** 2.2

---

### 2.6 Push-уведомления

**Описание:**
Настроить Expo Notifications:
- Регистрация устройства → сохранить `push_token` в профиле пользователя
- Celery задача: за час до каждого приёма пищи → push с названием контейнера
- Celery задача: за 12 часов до приёма → push если нужно переложить контейнер
- Celery beat: ежедневно в 8:00 → push с планом на день

**Результат:**
- Пользователь получает пуши о текущем контейнере и задачах подготовки
- Уведомления приходят по расписанию приёмов пищи из профиля

**Зависимости:** 2.5

## Фаза 4 — Полировка и запуск

**Цель:** Продукт готов к реальным пользователям.

---

### 4.1 Офлайн-режим

**Описание:**
Настроить React Query для работы без интернета:
- `persistQueryClient` с `AsyncStorage` — кэш сохраняется между сессиями
- Оптимистичные обновления для "Съел" и чекбоксов покупок
- Индикатор офлайн-режима при потере соединения
- Синхронизация при восстановлении

**Результат:**
- Главный экран, покупки, готовка — работают без интернета
- Изменения в офлайне синхронизируются при подключении

**Зависимости:** 3.3 (все экраны подключены к API)

---

### 4.2 Экран "Прогресс"

**Описание:**
Создать новый экран аналитики:
- График КБЖУ по дням (план vs факт)
- Соблюдение плана: % дней без отклонений
- Динамика веса (если пользователь вводит)
- Стрики и достижения

`POST /api/v1/progress/log` — логировать вес и факт приёма.

**Результат:**
- Пользователь видит динамику за 4 недели
- Агент использует данные прогресса для персонализации

**Зависимости:** 3.3

---

### 4.3 Тесты

**Описание:**
Написать тесты на критические пути:
- `nutri_service`: формулы BMR/TDEE для разных параметров
- `meal_planner_service`: КБЖУ сгенерированного плана в пределах нормы
- `POST /auth/register` + `POST /auth/login`: интеграционный
- `POST /plan/generate` → `GET /plan/{id}`: интеграционный
- Агент: unit-тесты на каждый tool

**Результат:**
- `pytest` проходит без ошибок
- Покрытие критических сервисов > 80%

**Зависимости:** 4.1

---

### 4.4 Production deploy

**Описание:**
Подготовить production-окружение:
- `docker-compose.yml` с полным стеком: api (2 реплики), worker, scheduler, db, redis, nginx
- `.env.production` — переменные окружения, секреты
- Nginx: SSL (Let's Encrypt), rate limiting, gzip
